# [BUG]  🐞Agent does not actually invoke tools, only simulates tool usage with fabricated output

**Issue #3154** | State: closed | Created: 2025-07-14 | Updated: 2026-03-13
**Author:** redvelvets
**Labels:** bug, no-issue-activity

### Description

When using a custom LLM together with a `WebSearchTool` and `Phoenix memory`, the agent **does not actually invoke the tool at runtime**, even though it produces a valid-looking `Thought → Action → Observation → Final Answer` trace.

Instead of executing the tool (e.g., calling `tool.run()`), the LLM **generates a fake Observation output** and continues to the final answer. There are no traces of tool usage in logs or the Phoenix interface. The behavior breaks the tool-use promise and leads to silent failures.

### Steps to Reproduce

1. Set up an agent and a task with a custom LLM and WebSearchTool:

   ```python
   from crewai import Agent, Task, Crew, Process, LLM
   from explore.web_search_tool import WebSearchTool
   from explore.phoenix_provider import register_phoenix_provider

   register_phoenix_provider(project_name="crewai-memory")

   web_search = WebSearchTool()
   llm = LLM(
       model="openai/DIFY-Qwen2.5-72B-Instruct-GPTQ-Int4",
       base_url="https://ai.auto.xiaomiev.com/v1",
       api_key="..."
   )

   agent = Agent(
       role="Research Expert",
       goal="Investigate and analyze trends in {topic}",
       llm=llm,
       tools=[web_search],
       verbose=True,
   )

   task = Task(
       description="Please research the latest developments in {topic}, including challenges and potential.",
       expected_output="A well-structured research report.",
       agent=agent,
       tools=[web_search],
   )

   crew = Crew(agents=[agent], tasks=[task], process=Process.sequential)
   result = crew.kickoff(inputs={"topic": "AI in Healthcare"})
   print(result)
   ```

2. Observe the logs and Phoenix interface:

   * The output includes a full `Thought`, `Action`, `Action Input`, `Observation`, and `Final Answer`.
   * However, the `WebSearchTool` is **never actually invoked**.
   * The `Observation` is fully generated by the LLM without calling `tool.run()`.

### Expected behavior

* When an agent emits `Action: Web Search`, the tool must be **actually invoked**.
* The observation should reflect the tool's real output, not fabricated content.
* Phoenix UI and memory traces should show the tool execution step.

**mouramax:**
@redvelvets, your organized and clear example (kudos for that) helps to better highlight one or two points that, IMHO, aren't as robust as they should be.

Have you ever had to fill out a form and ended up filling in a field you weren't supposed to? Meaning, the form had fields meant for someone else, but you filled them out because it wasn't clear? It's happened to me, and it left me wondering if the problem was with the person who designed the unclear form, with me for not understanding, or if we were both to blame. So, you receive a form that says:

```
You ONLY have access to the following tools, and should NEVER make up
tools that are not listed here:

Tool Name: Tool XYZ
Tool Arguments: {"arg1": {"description": "Description of arg1", "type": "string"}}
Tool Description: This tool does XYZ.

IMPORTANT: Use the following format in your response:

Thought: you should always think about what to do
Action: the action to take, only one name of [Tool XYZ], just the name,
exactly as it's written.
Action Input: the input to the action, just a simple JSON object, enclosed in
curly braces, using " to wrap keys and values.
Observation: the result of the action

Once all necessary information is gathered, return the following format:

Thought: I now know the final answer
Final Answer: the final answer to the original input question
```

How easy (or how hard) is it to understand that "Observation:" is something that should be filled in by the system, not by me? And if I go ahead and fill in "Observation:", have I not now "gathered all the necessary information"? Oh, so I can just reply with "Final Answer:"! This is the first point of fragility, in my opinion (and I want to say this a thousand times: this is solely and exclusively my opinion): the communication here definitely should be reviewed and improved.

If you look at the [parser's code](https://github.com/crewAIInc/crewAI/blob/6ebb6c9b63a4c1e9073a75287dda06edf2e7a643/src/crewai/agents/parser.py#L87), you'll see that the system can even protect itself against "Observation:" hallucinations. So if the LLM mistakenly responds with "Thought:", "Action:", "Action Input:", and "Observation:", CrewAI will only consider "Action:" and "Action Input:" and will make the call to the function/tool. However, if the LLM responds with "Final Answer:", CrewAI gives "Final Answer:" top priority and ends the task.

So, it's like you go to a restaurant, the waiter hands you the menu, and you immediately say, "Now that I've eaten a delicious pizza and paid the bill, it's time for me to leave." And the framework simply does nothing about it. And for me, this is the second fragility: the parser could implement something like a state machine and evaluate that it makes no sense for the LLM to respond with "Observation:" and "Final Answer:" without even having executed an action (at this point, the agentic loop knows that the LLM hasn't called any function/tool yet, and therefore, there can't be an "Observation:").

If you want to take your experiment a step further, add something like this to your `Task.description`:

```
Do NOT provide the 'Observation:' field yourself;
it will be provided for you after the tool runs.
```

Try running the same task 10 times without this instruction and 10 times with it. If there's any improvement in the quality of the responses, the rule is: **value your prompt**. Talk to your LLM. Give it examples. Provide unequivocal instructions. This will allow you to be less dependent on what a framework does (or doesn't do) behind the scenes.

**mouramax:**
@in-principio, from my understanding, this happens because CrewAI doesn't use function/tool calling as it's [implemented in LiteLLM](https://docs.litellm.ai/docs/completion/function_call), which is why you're not getting anything in the `tools` parameter.

CrewAI re-implements its own version of function/tool calling by essentially "chatting" with the LLM. It provides a list of defined functions and then parses the LLM's response to capture the intent of the function call. Then, CrewAI's agentic loop mechanism actually executes the action and appends the result of the execution to the messages queue. In other words, the entire process of `Listing Tools → Receiving the tool usage request → Executing the tool/function → Injecting the action's result` is handled internally by CrewAI. That's why you're not getting anything in the `tools` parameter.

This was a design decision that gives the framework the advantage of having full control over the mechanism. However, in my view, this means you have to be very careful with this communication layer with the LLM, to ensure it unequivocally understands when and how to call tools. And this is precisely where the **first weakness** I reported above lies. When you reinvent the wheel, it needs to be at least on par with the original, and it seems to me there's significant room for improvement in this communication mechanism within CrewAI.

This communication weakness is easily seen in the number of complaints pointing out the difficulty smaller models have in using tools/functions reliably. The OP is reporting difficulty with a 72B parameter model, which can hardly be considered small!

To wrap up my two cents, add to this the fact that other parts of the system depend directly on a solid tool implementation. As I've [already detailed here](https://community.crewai.com/t/does-hierarchical-process-even-work-your-experience-is-highly-appreciated/2690/8), the hierarchical process in CrewAI is heavily based on tool usage, which demands even greater care to ensure clear communication between the LLM and the framework.
