# Importing CSV and Parquet Files in DuckDB

DuckDB's best import feature is that it often doesn't look like import at all. For the common case, you point SQL at a file and start querying:

```sql
SELECT * FROM 'myfile.csv';
SELECT * FROM 'myfile.parquet';
FROM sniff_csv('lineitem_sample.csv');
COPY lineitem FROM 'lineitem_sample.csv' (FORMAT CSV, AUTO_DETECT TRUE);
```

The project README uses that exact file-in-`FROM` style, and the implementation backs it up. CSV paths flow through `read_csv_auto`, while Parquet paths resolve to `parquet_scan`. That's a sharp design choice. We don't need a loader service just to answer one question, and we don't need to invent a staging table before the first query becomes useful.

CSV is where the real engineering starts. Parquet carries schema in the file. CSV doesn't. DuckDB compensates with a sniffer that works out delimiter, quoting, escaping, header presence, date and timestamp formats, and candidate column types before the scan runs. And `sniff_csv()` is the underrated tool here. It doesn't just inspect a file. It returns the concrete `read_csv(..., auto_detect=false, ...)` call DuckDB would use, which means you can turn an inferred import into a checked-in contract. In the project tests, `lineitem_sample.csv` is sniffed into a pipe-delimited read with sixteen typed columns and `dateformat='%Y-%m-%d'`. That's exactly the kind of output you want before a one-off query turns into a pipeline.

I'd use that split deliberately. Auto-detect is simpler, but an explicit `read_csv` call gives you reproducibility. That's the better trade once the file becomes part of a scheduled job or a shared workflow. DuckDB's binder enforces the same distinction. If you call `read_csv` without auto-detection, you must provide columns, otherwise binding fails and tells you to use `read_csv_auto` or turn `AUTO_DETECT=TRUE` back on. So the system pushes exploration and production into two different modes, which is the right call.

Sampling matters more than most teams expect. In DuckDB's CSV tests for `issue_811.csv`, `SAMPLE_SIZE=1` leaves a sparse numeric column typed as `VARCHAR`, while `SAMPLE_SIZE=-1` promotes that same column to `DOUBLE`. So the sniffer is smart, but it isn't clairvoyant. If a feed has rare values, dirty tail rows, or columns that only become informative late in the file, the sampling budget changes the inferred schema. The implementation makes that behavior explicit too: `sample_size=-1` means "read the whole thing" during sniffing. That's slower up front. But it's often cheaper than debugging a bad type after the data has already moved downstream.

Multi-file ingestion is where DuckDB stops feeling like a notebook convenience and starts looking like a backend primitive. The shared multi-file reader exposes `union_by_name`, `filename`, and `hive_partitioning` across formats. For Parquet, the tests show `parquet_scan('__TEST_DIR__/union_by_name_lineitem*.parquet', union_by_name=TRUE)` merging files whose columns were written in different orders. CSV gets the same idea, but with more work under the hood. Schema discovery keeps sniffing files until it reaches a row budget, then merges the discovered schemas into one view of the dataset. We should treat `union_by_name` as the right answer when files evolve over time. We shouldn't pretend it removes the cost of schema drift.

Parquet is cleaner because the file format does more of the job for you. DuckDB tests cover both explicit `parquet_scan(...)` and the shorter file-path form in `FROM`, and both are first-class. The benchmark setup in `benchmark/ingestion/load_tpch.sql` and `benchmark/large/ingestion/tpch/load_tpch_sf10.sql` makes the pattern obvious: generate TPC-H data, export it to `duckdb_benchmark_data/tpch_csv` and `duckdb_benchmark_data/tpch_parquet`, then create parallel `*_csv` and `*_parquet` views over those files. That's a strong signal from the project itself. DuckDB expects engineers to work directly against external CSV and Parquet, not to immediately convert everything into native storage.

But file scans and imports are different tools. Reading a file in `FROM` is the simplest path, but `COPY` gives you a real table with a fixed schema and stable downstream behavior. The CSV round-trip tests make this concrete: `COPY lineitem FROM '{DATA_DIR}/csv/real/lineitem_sample.csv' DELIMITER '|'` loads the table, and `COPY lineitem TO '{TEMP_DIR}/lineitem.csv' (DELIMITER ' ', HEADER)` writes it back out. There's also a pragmatic middle ground for messy CSV. `COPY ... (FORMAT CSV, AUTO_DETECT TRUE)` lets you land the data without spelling out every parsing option on day one. So my rule is simple. Query files directly for ad hoc analysis and thin ingestion layers. Use `COPY` when the data becomes part of your model, needs stable column types, or will be joined enough times that reparsing the file stops being clever.

We do need to be honest about the rough edges. Remote CSV isn't the same as local CSV. Issue `#21088`, created on 2026-02-26, shows that in DuckDB `1.4.4 6ddac802ff`, a remote `https://cmip6-pds.s3.amazonaws.com/pangeo-cmip6.csv` object with `Content-Length: 83974211` did not populate the external file cache at all. `duckdb_external_file_cache()` stayed empty, and `EXPLAIN ANALYZE` showed roughly `80.0 MiB` of HTTP input for the scan. That's a real operational consideration if you're validating or re-querying the same CSV on S3. Parquet has its own caveat. An open `1.5.0` issue reports hive-partitioned Parquet on S3 discovering all files before pruning when recursive globbing is enabled, and the documented workaround is `SET s3_allow_recursive_globbing = false;`. So yes, Parquet is usually the easier format to import, but remote layout and partition pruning still matter.

That's the real takeaway. DuckDB makes import look like query planning, and that's mostly a good thing. We can start with a file path, graduate to `sniff_csv()` when CSV gets weird, add `union_by_name` when schemas drift, and only land data with `COPY` when the workload justifies it. I like that model because it keeps the easy path short without pretending text files are self-describing. And it gives us an escape hatch long before "just build another ingestion service" becomes the default answer.
