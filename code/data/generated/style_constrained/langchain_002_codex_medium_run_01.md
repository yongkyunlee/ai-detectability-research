# Building RAG Pipelines with LangChain Retrievers and Vector Stores

Most RAG discussions start at the vector database. LangChain’s codebase starts one layer higher, and that’s the better design lens. In `langchain-core`, a vector store stores embedded data and runs vector search. A retriever is more general. It only has to return `Document` objects for a text query. That distinction matters because it keeps storage, ranking, and answer generation from collapsing into one opaque box.

So the clean mental model is straightforward. Documents get loaded and split. Embeddings turn text into vectors. A vector store persists those vectors and exposes search. Then a retriever decides what the generation step should actually see. In LangChain, the handoff is usually `vector_store.as_retriever(...)`, which wraps a store in the standard `BaseRetriever` interface. That sounds small. It isn’t. Once everything is a retriever, you can swap a plain similarity lookup for query rewriting, parent-document lookup, compression, or rank fusion without rewriting the rest of the pipeline.

The default retriever path is intentionally boring, and that’s a good thing. `VectorStore.as_retriever` supports three search modes out of the box: plain similarity, maximal marginal relevance with `search_type="mmr"`, and thresholded retrieval with `search_type="similarity_score_threshold"`. I like this API because it exposes the real trade-off instead of hiding it behind marketing names. Similarity is the simplest choice, but MMR gives you diversity when your corpus has many near-duplicate chunks, and score-threshold retrieval is useful when “return nothing” is better than returning weak context. We shouldn’t pretend those modes are interchangeable. They optimize for different failure cases.

LangChain’s newer chain composition also reflects that separation. `create_retrieval_chain` wires a retriever to a document-combining runnable, and `create_stuff_documents_chain` handles the simplest answer step by formatting retrieved documents into a single prompt variable, usually `context`. That is a better shape than older all-in-one helpers because you can see the seams. The older `RetrievalQA` path still exists in `langchain-classic`, but the code marks it as deprecated in favor of `create_retrieval_chain`. We should follow that lead.

Vector store choice still matters, of course. The LangChain partner packages make that explicit. In this repo, both `langchain-chroma` and `langchain-qdrant` are version `1.1.0`, and both are published as separate integration packages rather than being buried inside the main package. Chroma is the easy local-onramp. The docs show `pip install -qU chromadb langchain-chroma`, then a `Chroma(...)` store with a collection name, embedding function, metadata filters, similarity search, score search, and MMR. If your first milestone is “get documents indexed and retrieved with minimal ceremony,” Chroma is hard to beat.

But Qdrant clearly targets a more opinionated retrieval stack. The `QdrantVectorStore` class exposes `RetrievalMode.DENSE`, `SPARSE`, and `HYBRID`, and the package includes `FastEmbedSparse` plus an optional `fastembed` dependency. That means the store can participate in sparse and hybrid retrieval directly. The trade-off is complexity. Chroma is simpler, but Qdrant gives you typed filters, multiple retrieval modes, and a clearer path to hybrid retrieval. The issue tracker even shows one example of that extra surface area: a documentation fix was needed because Qdrant filtering worked with `from qdrant_client import models`, not the older `qdrant_client.http` import shown in docs.

I wouldn’t stop at `as_retriever()` once the first relevance problems show up. LangChain already ships specialized retrievers for the common second-order failures. `ParentDocumentRetriever` is a strong example. Its own docstring spells out the tension: small chunks produce better embeddings, but larger chunks preserve context. The retriever resolves that by indexing child chunks while returning parent documents from a docstore. `MultiVectorRetriever` uses the same general pattern and returns parent documents after searching over embedded children. That is often a better RAG shape than shoving tiny fragments directly into generation and hoping the model reconstructs the larger idea.

And query handling matters just as much as chunking. `MultiQueryRetriever` uses an LLM to generate three alternate versions of the user question, then unions the retrieved results. `ContextualCompressionRetriever` runs a base retriever first and compresses the results afterward. `create_history_aware_retriever` only rewrites the query when `chat_history` exists, which is exactly what a chat RAG system should do. We don’t need to make retrieval smarter everywhere. We need to make it smarter at the point where it fails. That’s a much cheaper mindset than reaching for an elaborate hybrid stack on day one.

The LangChain issues and community threads are useful here because they show where real teams get cut. A January 12, 2026 Reddit post about getting started with RAG described the ecosystem as overwhelming because of all the moving parts and the amount of deprecated syntax in tutorials. The repo history backs that up. An October 6, 2025 issue reported `ModuleNotFoundError: No module named 'langchain.retrievers'` when trying to import `ContextualCompressionRetriever` from the wrong package in the v1 era. The answer was not “don’t use compression.” The answer was “import it from `langchain_classic.retrievers`.” Another documentation issue showed `create_retriever_tool` had moved to `langchain_core.tools`. Package boundaries are not trivia here. They are part of the architecture.

There’s also a practical lesson in the bugs. One Chroma issue showed document counts climbing from `6` to `12`, `18`, `24`, `30`, and `36` across iterations when the caller expected a fresh in-memory store each time. That is a great reminder that retrieval bugs are often state bugs. LangChain’s own `langchain-tests` package leans into that reality with `VectorStoreIntegrationTests`, which check basic behaviors like empty-store initialization, add/delete correctness, and stable document IDs. We should copy that attitude in application code. Retrieval quality is not only about better embeddings. It’s also about lifecycle hygiene, deterministic interfaces, and verifying that your store is actually empty when you think it is.

That’s the real takeaway from LangChain’s retrieval stack. Keep the pipeline modular. Treat the retriever as the control plane, not just a convenience wrapper around a database call. Start with a plain vector store and `as_retriever`, then add specialized retrievers only when the failure mode is obvious. And keep one eye on package boundaries, because in this codebase the difference between `langchain-core`, `langchain-classic`, and a partner package like `langchain-qdrant` or `langchain-chroma` is not cosmetic.
