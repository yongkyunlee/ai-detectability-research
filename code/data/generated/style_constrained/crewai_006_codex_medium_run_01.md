# Memory and Context Management in CrewAI

CrewAI treats memory and context as related but different problems. Persistent memory is about what an agent should be able to recall later. Context management is about what can safely fit into the next prompt without blowing the model's token budget.

As of CrewAI `1.11.0`, the framework has collapsed several older concepts into one unified `Memory` API. The docs describe it as a single class replacing separate short-term, long-term, entity, and external memory types. `Memory` defaults to LanceDB-backed storage, lazy-initializes its model and embedder, and exposes one surface for `remember()`, `remember_many()`, `recall()`, `forget()`, `scope()`, and `slice()`.

That unification matters because most agent frameworks get messy at the exact point where “memory” stops meaning “whatever is still in the prompt.” CrewAI doesn't make that mistake. With `memory=True`, a crew builds a default `Memory()` instance and passes the crew embedder through automatically. If you need control, you can pass a configured `Memory` object instead. The explicit path is better when you care about isolation boundaries.

The storage defaults are opinionated in a sensible way. Memory is backed by LanceDB and stored under `./.crewai/memory` unless you override the path or set `CREWAI_STORAGE_DIR`. The default embedder matches OpenAI’s `text-embedding-3-small`, which is why the LanceDB backend assumes a 1536-dimensional vector unless it can infer something else from the table. The default LLM for analysis is `gpt-4o-mini`.

But retrieval quality in CrewAI doesn't come from vector similarity alone. The ranking formula is explicit: semantic similarity, recency decay, and record importance are blended into one composite score. By default, the weights are 0.5 for semantic relevance, 0.3 for recency, and 0.2 for importance, with a 30-day recency half-life. That's a better default for engineering work than pure cosine similarity because old but important decisions shouldn't disappear just because a newer memory is nearby in embedding space.

So the real design win is scope. CrewAI organizes memory as a hierarchy of paths like `/project/alpha`, `/agent/researcher`, or `/customer/acme-corp`. That sounds small, but it's the difference between retrieval and rummaging. A scoped memory view restricts reads and writes to one branch. A slice can merge several branches into one read surface, and it can be read-only. For multi-agent backends, that's exactly the control plane you need. Shared crew memory is convenient, but scoped memory gives you something more valuable: predictable context boundaries.

CrewAI also separates persistence from write latency. `remember_many()` is non-blocking and saves in a background thread, while `recall()` acts as a read barrier and drains pending writes before it searches. That's a practical backend choice. If you're running Flows, the same model shows up there too. Flows get `self.remember()`, `self.recall()`, and `self.extract_memories()` out of the box, and the docs say that memory persists across runs. Since version `0.103.0`, `crewai run` is the recommended way to execute flows from the CLI.

Deep recall is where CrewAI gets more ambitious. `depth="shallow"` is just vector search plus composite scoring, and the docs peg it at roughly 200 ms with no LLM call. `depth="deep"` runs a RecallFlow that analyzes the query, suggests scopes, performs parallel searches, and can explore again when confidence is low. Short queries under the default `query_analysis_threshold` of 200 characters skip that LLM analysis entirely, which the docs say saves about 1 to 3 seconds per recall. Shallow recall is simpler, but deep recall gives you better routing when the query is broad, temporal, or badly phrased. That's a real trade-off, not a marketing checkbox.

Now the harder part: live prompt context. CrewAI has context-window management, but it lives on a different axis than memory. Agents default `respect_context_window=True`, and the compaction path in `agent_utils.py` summarizes non-system messages, preserves system messages, and rebuilds the conversation as a structured summary when the model limit is exceeded. That's prompt compaction, not long-term memory. If you don't keep those concepts separate, you'll end up debugging retrieval when the real problem is prompt growth.

And CrewAI's issue tracker shows why that distinction matters. A closed issue from February 5, 2026 documented that Anthropic context-limit errors were not recognized for automatic summarization because the detector in `crewai/utilities/exceptions/context_window_exceeding_exception.py` only matched OpenAI-style patterns. The example error was blunt: `"prompt is too long: 210094 tokens > 200000 maximum"`. Another issue, opened on February 6, 2026 against CrewAI `1.9.3`, showed task-state bleed when `CrewAgentExecutor` reused messages and iteration counters between sequential tasks. That's not a memory bug. That's a context-boundary bug. A March 1, 2026 feature request points at a third gap: inter-task context in `Crew._get_context()` still aggregates prior task outputs verbatim through `aggregate_raw_outputs_from_task_outputs()`, which means raw output can snowball even if your persistent memory is clean.

We should read those issues as design signals. CrewAI's memory layer is getting more structured. Its context assembly path is still more stringly and more fragile. That's common in agent frameworks. Retrieval gets architectural attention. Prompt assembly grows organically until it hurts.

The community reaction is useful here. One March 6, 2026 Reddit post described CrewAI's built-in memory as a black box and promoted a markdown-based alternative with files like `SOUL.md` and `MEMORY.md`. A Hacker News thread from the same day made the opposite argument in broader agent terms: the simplest persistence often wins, and explicit text can outperform fancy retrieval when humans need to inspect or edit state. I wouldn't take either argument as universal. But I would take the pressure seriously. Built-in memory is simpler, but explicit and inspectable memory gives you operational leverage when behavior drifts.

So if I were setting a team standard for CrewAI, it would be this. Use unified memory, but treat scope design as part of your system design, not a documentation detail. Use `memory=True` for prototypes, then graduate to an explicit `Memory(...)` configuration once you know what should persist, where it should live, and who should see it. Use deep recall for messy synthesis queries, and shallow recall for tight operational lookups. And watch prompt context separately from memory health, because CrewAI does not currently make those failure modes disappear just because both happen to involve “context.”
