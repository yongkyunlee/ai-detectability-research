## Query Performance Optimization in DuckDB

DuckDB performance tuning starts with a blunt reality: the engine is usually fast when it scans less data, not when you throw more hardware at a bad plan. In `src/optimizer/optimizer.cpp`, DuckDB runs rewrites for CTE inlining, filter pushdown, join ordering, unused-column removal, and column lifetime analysis. The optimizer list in `src/common/enums/optimizer_type.cpp` is equally revealing. Names like `filter_pushdown`, `unused_columns`, `row_group_pruner`, `sampling_pushdown`, `materialized_cte`, and `top_n_window_elimination` tell you what the engine is trying to do on your behalf.

So the first optimization rule is simple: verify that DuckDB is actually doing those things for your query shape. Don’t tune from intuition. Tune from the plan.

DuckDB gives you enough instrumentation to do that without guesswork. `EXPLAIN ANALYZE` is the first stop, especially for file-backed queries where the plan can expose file counts and HTTP activity. If that’s not enough, the built-in profiler can emit JSON with operator and scan metrics. The test suite uses commands like `PRAGMA enable_profiling = 'json';`, `PRAGMA profiling_output = 'profiling_output.json';`, and custom profiling settings for `OPERATOR_ROWS_SCANNED`, `CUMULATIVE_ROWS_SCANNED`, `TOTAL_BYTES_READ`, and optimizer timings.

We should be opinionated here: projection and pruning are the real performance features. Everything else is secondary. A closed DuckDB issue from February 2026 shows why. In issue #20958, a simple pass-through CTE referenced twice went from about 0.3 seconds on DuckDB 1.3.2 to 31.64 seconds on 1.4.4. Same logical result. Totally different physical behavior. The reproducer built a 50,000,000-row table with 50 columns, then selected only `id` and `value` from a `WITH base AS (SELECT * FROM wide_table)` CTE used twice in a `UNION ALL`. In 1.3.2, the CTE was inlined and only two columns were scanned. In 1.4.x, the CTE was materialized with all 50 columns, which blocked column pruning and blew up runtime.

That example matters because it kills a common misconception. Writing a query in a “cleaner” way doesn’t mean DuckDB gets a cleaner plan. Sometimes the opposite is true. A `MATERIALIZED` CTE is simpler when you need to pin a plan, but `NOT MATERIALIZED` or plain inlining gives DuckDB more room to prune columns and push filters. If a pass-through CTE exists only for readability, I wouldn’t let it become a permanent optimization barrier.

But local scans are the easy case. Remote Parquet is where plan shape really starts to matter. DuckDB issue #21347, opened on 2026-03-12, describes a regression in v1.5.0 (Variegata) 3a3967a for hive-partitioned Parquet on S3. With a filter that should touch about 39 matching files, `EXPLAIN ANALYZE` showed `Scanning Files: 39/39` on v1.4.4 but `Scanning Files: 39/122` on v1.5.0. Same answer set. Extra file discovery, extra footer reads, extra latency. The documented workaround was `SET s3_allow_recursive_globbing = false;`, which restored the older behavior.

And that wasn’t an isolated edge case. Issue #21348, also opened in March 2026, shows how a seemingly smart rewrite can backfire badly on remote storage. A query using `QUALIFY row_number() OVER (...) = 1` against wide, hive-partitioned Parquet files on S3 went from 81 HTTP GET requests and 11.6 seconds on v1.4.4 to 4,214 GET requests and 31.5 seconds on v1.5.0. The reported cause was DuckDB’s `top_n_window_elimination` rule, which rewrote the plan into an `arg_max` plus `HASH_GROUP_BY` and `SEMI JOIN` pattern that scanned the Parquet data twice. That’s often fine on local disk. It’s brutal over object storage when the files are wide and each extra pass turns into more range requests.

We also get the right lesson from the workaround. The narrow fix was `SET disabled_optimizers='top_n_window_elimination';`. The broader workaround was wrapping the scan in a `MATERIALIZED` CTE. The best result combined `MATERIALIZED` with `SET s3_allow_recursive_globbing = false;`, bringing the request count back down to 82 and wall-clock time to 11.5 seconds. That trade-off is worth stating plainly: `MATERIALIZED` is the simpler hammer, but disabling one optimizer or fixing file discovery gives you a more targeted correction.

And yes, hardware still matters. It just matters after the plan is sane. A community benchmark posted on 2026-01-27 used roughly 20 GB of ZSTD-compressed Parquet across 57 queries. On local storage, DuckDB’s warm median dropped from 881 ms at 8 threads and 16 GB RAM to 284 ms at 32 threads and 64 GB RAM. That’s a real gain. But the same discussion showed a 14 to 20 second cold-start penalty for DuckDB plus remote R2 storage because the first query had to fetch Parquet metadata, including file footers, schema, and row-group information. Window-heavy workloads were especially painful there: 947 ms for local DuckDB XL versus 12,187 ms for DuckDB + R2 XL.

So more threads help when the operator can parallelize and when the bottleneck is CPU work instead of remote request overhead. They don’t rescue a bad storage access pattern. A separate issue on sampling performance makes that point from another angle. In DuckDB 1.4.4, `BERNOULLI` and `SYSTEM` sampling did not scale with higher `SET threads` values, while `RESERVOIR` did. The underlying reason, discussed in the issue, was that seeded sampling disabled parallel operation in the streaming sampler. That’s a useful reminder for backend engineers: thread count is not a universal accelerator. It’s a multiplier on the execution paths that can actually use it.

The practical playbook is short. Start with `EXPLAIN ANALYZE`. Turn on JSON profiling when the plan alone isn’t enough. Keep projections narrow and make filters as literal and early as possible. Be skeptical of pass-through CTEs that materialize wide intermediates. On remote Parquet, watch file counts and request counts as closely as operator timing. And if a new release gets slower, inspect the specific rewrite before you rewrite the whole query. DuckDB exposes optimizer names through `duckdb_optimizers()` and lets you disable one rule at a time.

DuckDB is fast because the engine is aggressive. That’s good news. But aggressive optimizers are only an advantage when the query shape and storage layout give them the right target. Query performance work in DuckDB is mostly the discipline of making that target obvious.
