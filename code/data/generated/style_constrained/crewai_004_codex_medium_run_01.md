# Using tools in CrewAI agents

Tools are the line between a chatbot with ambition and an agent that can actually touch the world. CrewAI leans into that line pretty hard. The docs treat tools as first-class agent capabilities, and the code follows through with schema validation, caching, hook points, usage limits, and runtime events.

If you scaffold a project with `crewai create crew <your_project_name>`, the generated layout gives you `crew.py`, `config/agents.yaml`, `config/tasks.yaml`, and a `src/my_project/tools/` directory. That structure is right. Tooling belongs next to orchestration, because tool choice is part of the agent contract, not an afterthought hidden in a helper module.

CrewAI’s happy path is straightforward. Install the extra package with `pip install 'crewai[tools]'`, import something like `SerperDevTool` or `FileReadTool`, and attach it to an `Agent` with `tools=[...]`. The docs now advertise 40+ prebuilt tools across search, files, databases, scraping, cloud, and code execution.

For simple tools, the `@tool` decorator is the cleanest interface. You write a function, give it a docstring and type annotations, and CrewAI turns it into a callable tool with a generated argument schema. That’s a good default for internal utilities and narrow side effects. But subclassing `BaseTool` is the better choice once the tool matters operationally. It gives you an explicit `args_schema`, a `_run()` implementation, optional async support through `_arun()`, and room for behavior such as `cache_function`, `result_as_answer`, and `max_usage_count`.

The decorator is simpler, but `BaseTool` gives you a real surface for contracts and controls. If the tool talks to a billing API, mutates files, or becomes part of a production workflow, I wouldn’t hide it behind a lightweight function wrapper unless there’s a strong reason.

The implementation details make that judgment easy to defend. In `BaseTool`, CrewAI validates kwargs against Pydantic schemas before execution, increments usage counters atomically, and rejects over-limit calls with a clear guard. In the decorator path, the generated tool can still carry `result_as_answer=True` or `max_usage_count`, which is useful, but the class-based path is much easier to reason about when a tool becomes critical infrastructure rather than glue code.

There’s another design choice that matters more than people admit: where tool calling is modeled. CrewAI lets you set `function_calling_llm` either at the crew level or per agent. Crew-level configuration is simpler and cheaper when you want every agent to use the same model for tool invocation, and the docs even show a pattern using `gpt-4o-mini` for that. But agent-level override gives you sharper control when one agent needs a stronger calling model and another does not. Shared defaults are convenient. Per-agent control is safer when tool behavior differs materially across roles.

Once an agent starts running, CrewAI’s execution path is more serious than the tutorial examples suggest. The agent executor converts tools into structured schemas, asks the model for tool calls, parses arguments, checks cache, runs before hooks, executes the tool, runs after hooks, and emits lifecycle events such as `ToolUsageStartedEvent`, `ToolUsageFinishedEvent`, and `ToolUsageErrorEvent`. It means tool use is observable and interceptable, not just buried inside prompt text.

And the hooks are one of the better pieces of the design. `before_tool_call` hooks can mutate inputs in place, validate arguments, require approval, or block execution entirely by returning `False`. `after_tool_call` hooks can sanitize or transform the result before the agent sees it. The `ToolCallHookContext` includes the tool name, mutable input, tool instance, agent, task, crew, and result. So you can build real policy around tool use instead of pretending prompt wording is policy.

That said, don’t confuse a nice abstraction with a solved problem.

The project’s own issue history is a useful correction. Issue #3154, opened on July 14, 2025 against CrewAI `0.141.0`, describes an agent producing a convincing `Thought -> Action -> Observation -> Final Answer` trace without ever running the custom tool code. Issue #4788, opened on March 9, 2026 against `1.10.1`, describes native tool calls being discarded when a model returned assistant text alongside tool-call payloads. Those are bug reports, not a final verdict on the framework. But they point at the failure mode that matters: a plausible transcript is not proof of execution.

So if you’re using tools in CrewAI, verify tool behavior through events, logs, and traces. Don’t stop at “the model said it used the tool.” The event system and tool hooks exist for a reason. The community discussions in this corpus make the same point from a different angle: debugging agent systems often fails because teams patch the visible symptom first instead of checking routing, tool execution, memory injection, or delegation drift.

MCP-backed tools raise the stakes again. CrewAI now supports MCP directly on agents through the `mcps` field, and the simple form is refreshingly terse. You can point an agent at an HTTP endpoint or a catalog slug and get discovered tools automatically. For quick integration, that’s excellent. But the structured configuration path using `MCPServerStdio`, `MCPServerHTTP`, or `MCPServerSSE` is the one to prefer once authentication, headers, transport, or tool filtering start to matter. The string DSL is simpler, but structured MCP configuration gives you actual operational control.

We also shouldn’t talk about tool use without talking about trust. The MCP security docs are blunt: only connect to MCP servers you fully trust. They call out prompt injection through tool metadata, confused deputy attacks, DNS rebinding concerns, and the need for strict input validation. That warning is not theoretical. Separate security reports in this corpus pushed on `CodeInterpreterTool`, and the March 2026 changelog shows the project responding with fixes: `v1.11.0rc1` notes a sandbox escape fix, and `v1.11.0rc2` replaces `os.system` with `subprocess.run` in unsafe-mode package installation. It’s a reminder that tools are part of your security boundary.

My view is simple. In CrewAI, prompts decide intent, but tools carry side effects. Treat them like backend interfaces. Keep descriptions tight, schemas explicit, hooks cheap, observability on, and trust boundaries narrow. CrewAI gives you enough machinery to do that well. But you only get reliable agents if you use that machinery like infrastructure instead of demo code.
